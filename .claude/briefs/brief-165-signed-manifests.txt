### Background

When users install the kit from a tagged release, they should be able to verify
the integrity of what they installed. The plan:
1. On each GitHub Release, generate a `SHA256SUMS` file listing SHA-256 checksums
   of the installable assets
2. Sign it using sigstore/cosign (keyless signing, no GPG key management needed)
3. Document the verification steps in README and docs/RELEASE_CHECKLIST.md

This does NOT require an existing release tag to implement — we create the
workflow now and it will fire when the first tag is pushed.

### Files to create / change

#### 1. Create `.github/workflows/release-checksums.yml`

This workflow triggers on GitHub Release publication and generates SHA256SUMS.
Use sigstore/cosign for keyless signing (no GPG key setup needed).

```yaml
name: Release — checksums and signing

on:
  release:
    types: [published]

permissions:
  contents: write
  id-token: write

jobs:
  checksums:
    name: Generate SHA256SUMS and sign
    runs-on: ubuntu-latest
    steps:
      - name: Checkout at tag
        uses: actions/checkout@v4
        with:
          ref: ${{ github.event.release.tag_name }}

      - name: Collect installable assets
        run: |
          mkdir -p dist
          # Copy all user-facing installable files into dist/
          cp -r skills/ dist/skills
          cp -r tooling/ dist/tooling
          cp -r project-template/ dist/project-template
          cp scripts/install.sh scripts/install.ps1 \
             scripts/update.sh scripts/update.ps1 \
             scripts/uninstall.sh scripts/uninstall.ps1 \
             scripts/validate.sh scripts/validate.ps1 \
             scripts/new-skill.sh scripts/new-skill.ps1 \
             scripts/doctor.sh scripts/doctor.ps1 \
             dist/
          cp VERSION .kit-manifest dist/

      - name: Generate SHA256SUMS
        run: |
          cd dist
          find . -type f | sort | xargs sha256sum > ../SHA256SUMS
          cat ../SHA256SUMS

      - name: Install cosign
        uses: sigstore/cosign-installer@v3

      - name: Sign SHA256SUMS with cosign (keyless)
        run: |
          cosign sign-blob --yes SHA256SUMS \
            --bundle SHA256SUMS.bundle

      - name: Upload SHA256SUMS and bundle to release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            SHA256SUMS
            SHA256SUMS.bundle
```

**Note on action versions**: Look at other workflow files in `.github/workflows/`
to match the `actions/checkout` version convention used in this repo. Use the
same version (tag or SHA-pinned) for consistency.

**Important**: The `softprops/action-gh-release` action is used for uploading
release assets. Check if a similar action is already used elsewhere in the
repo — if so, use the same version. If not, use `@v2`.

So the verification section in the file should be:
```markdown
## Verification

Users can verify the integrity of a release using `SHA256SUMS` published on the Releases page.

```bash
# Verify a specific file (e.g. scripts/install.sh)
sha256sum --check --ignore-missing SHA256SUMS

# Verify the signature with cosign (keyless)
cosign verify-blob SHA256SUMS --bundle SHA256SUMS.bundle \
  --certificate-identity-regexp ".*" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com"
```

`SHA256SUMS` and `SHA256SUMS.bundle` are automatically generated and uploaded by
the `release-checksums.yml` CI workflow when a release is published.
```

### Constraints

- Do not modify `tooling/`, `scripts/`, `skills/`, `project-template/`, or
  any non-CI/docs files.
- No new npm/pip/cargo dependencies (cosign is installed via sigstore/cosign-installer action).
- Cosign signing must be keyless (no GPG keys) — use `--yes` flag to avoid prompts.
- MIT-compatible licenses only (sigstore/cosign is Apache-2.0, fine).
- The workflow file should use `permissions: contents: write` (to upload release assets)
  and `id-token: write` (for OIDC token for keyless signing).
