### Context

This is the ai-agent-kit — a versioned config kit for Claude Code, Codex CLI,
and Antigravity CLI. It ships skill files, agent configs, and install scripts so
engineering teams can bootstrap AI coding assistants consistently.

The kit currently routes skills via a static keyword table in the main router
files (CLAUDE.md, AGENTS.md, AGY.md). Agents load every listed skill by name
when one happens to match — there is no intent classification, no lazy-loading of
sub-references, and no principled delegation decision.

Phase 4 of the roadmap introduces adaptive routing: the agent should infer task
intent and activate only the skills and subagents actually useful for the task,
then load references inside a skill only when signals justify it.

This issue (#414) is the foundation. It produces one design document. Do not
implement code, do not modify skill files, do not add scripts.

---

### Goal

Create `docs/ai/ADAPTIVE_ROUTING.md` — a clear, concrete design for adaptive
skill, subskill/reference, and subagent routing.

### What the document must contain

#### 1. Purpose section (5–10 lines)
Why adaptive routing exists: the static table loads too much context for simple
tasks and not enough for complex multi-area tasks. Adaptive routing saves tokens
for simple work and activates depth for complex work.

#### 2. Routing pipeline (the five stages)

Define a 5-stage pipeline. Each stage must have a name, a one-sentence
description, inputs, and outputs.

Stage 1 — **Classify task intent**
- Input: task description text (free text), optional list of changed/mentioned
  file paths
- Method: keyword matching + file extension patterns (no LLM call)
- Output: one or more intent labels from this fixed vocabulary:
  `review`, `implement`, `fix`, `refactor`, `docs`, `ci`, `security`,
  `data-migration`, `small-change`
- Rule: if no label matches with confidence, classify as `small-change`
  (fallback → do less)

Stage 2 — **Select skills**
- Input: intent labels, file paths (globs), task keywords
- Method: match against skill frontmatter fields: `task_intents`, `paths`
  (file globs), `keywords`
- Output: a small candidate set (default cap: 4 skills)
- Rule: a skill is selected only when at least one strong signal matches
  (file glob OR exact keyword OR explicit task_intent match). Weak or
  speculative matches do not load a skill.

Stage 3 — **Select references/subskills**
- Input: selected skills, task description, file paths
- Method: for each selected skill, check the `load_when` condition on each
  reference file in that skill's `references/` directory
- Output: list of reference files to load (may be empty)
- Rule: if no `load_when` condition is met, do not load any reference beyond
  the base SKILL.md

Stage 4 — **Decide delegation**
- Input: selected skills, intent labels, task scope estimate (number of
  independent technical areas)
- Decision criteria:
  - Delegate when ALL of:
    - Task involves 2 or more independent technical areas (e.g. backend + frontend)
    - Total context estimate would exceed ~50% of the context window
    - The areas can be reviewed or worked independently
  - Do NOT delegate when:
    - Task is a single-file change
    - Task is docs-only, CI-only, or formatting-only
    - User has explicitly scoped the task to one area
    - Only 1 skill is selected
  - Default cap: 2–3 subagents maximum
- Output: a delegation plan (YAML shape shown in examples below) or "no delegation"

Stage 5 — **Synthesize results**
- Input: subagent outputs (concise structured summaries)
- The main agent: deduplicates findings, resolves conflicts between subagent
  reports, proposes a unified next action, and presents a single coherent answer
- Rule: subagents return summaries, never full transcripts

#### 3. Context protection rules

List these rules explicitly:
- Load SKILL.md only (not SKILL.deep.md, not references/) unless signals justify depth
- A strong signal = matching file glob OR explicit keyword in task text
- Never load a skill speculatively ("this might be relevant")
- A skill not selected at Stage 2 is not loaded, even partially
- The context budget for skill content should stay below 20% of the context window
  for a typical task
- The main agent must be able to state which signal triggered each skill load

#### 4. Fallback behavior

When intent classification returns low confidence (nothing matches clearly):
- Default to `small-change` intent
- Load zero extra skills
- Offer: "I can load additional context for [area] if you confirm that's needed"
- Never preload context just in case

#### 5. Explainability

The agent must be able to produce a one-line reason for each selection:
- "Loaded dotnet skill: `src/Domain/Trip.cs` matched the dotnet `**/*.cs` path
  pattern"
- "Loaded code-review skill: task text contains the keyword 'review'"
- "Loaded ddd-cqrs reference: `src/Domain/` path matched the `**/Domain/**` load
  condition"
- "No delegation: task involves one technical area (dotnet backend only)"

#### 6. When NOT to activate extra context

Be explicit about cases where no extra context loads:
- Typo fixes, formatting, doc wording changes
- Single-file changes where the skill is obvious from the file type alone
- When the user's description already fully constrains the scope
- When the user explicitly says "quick fix" or "small change"
- When `small-change` intent is classified and no strong secondary signal exists

#### 7. Four worked examples

Present as clear before/after routing tables.

**Example A — Full-stack review**

Task: "Review and fix my Planora project"
Files hint: `src/Api/Program.cs`, `src/Domain/Trip.cs`, `apps/web/src/app/app.component.ts`

| Stage | Result |
|---|---|
| Intent | `review` |
| Skills selected | `code-review` (keyword "review"), `dotnet` (*.cs files), `angular` (*.ts + apps/web path) |
| References loaded | `code-review/references/architecture-review.md` (review + .cs Domain path), `dotnet/references/ddd-cqrs.md` (Domain path) |
| Delegation | yes — backend (.cs) and frontend (.ts/Angular) are independent; 2 subagents |
| Delegation plan | `backend-reviewer` (scope: Api, Domain, Application; skills: code-review, dotnet) + `frontend-reviewer` (scope: apps/web; skills: code-review, angular) |
| Synthesis | main agent merges findings, deduplicates, proposes unified next actions |

**Example B — .NET DDD backend only**

Task: "Add the TripApproval aggregate to the Domain layer"
Files hint: `src/Domain/`, `src/Application/`

| Stage | Result |
|---|---|
| Intent | `implement` |
| Skills selected | `dotnet` (*.cs path match + keywords dotnet/domain) |
| References loaded | `dotnet/references/ddd-cqrs.md` (`**/Domain/**` path match) |
| Delegation | no — single technical area |

**Example C — Typo fix**

Task: "Fix typo in README.md"
Files hint: `README.md`

| Stage | Result |
|---|---|
| Intent | `small-change` (fallback — no strong match) |
| Skills selected | none |
| References loaded | none |
| Delegation | no |

**Example D — CI workflow change**

Task: "Set up GitHub Actions workflow for the release pipeline"
Files hint: `.github/workflows/`

| Stage | Result |
|---|---|
| Intent | `ci` |
| Skills selected | `github-workflow` (intent `ci` matches, `.github/` path match) |
| References loaded | none (no load_when conditions met for single-area CI task) |
| Delegation | no — single technical area |

#### 8. Delegation plan YAML shape

Show the canonical shape a delegation plan takes:

```yaml
should_delegate: true
reason: "backend (.cs, Domain) and frontend (.ts, Angular) can be reviewed independently"
subagents:
  - name: backend-reviewer
    scope: "src/Api, src/Domain, src/Application, src/Infrastructure"
    skills: [code-review, dotnet, architecture]
    intent: review
  - name: frontend-reviewer
    scope: "apps/web/src"
    skills: [code-review, angular]
    intent: review
merge_strategy: >
  Main agent synthesizes findings: deduplicates issues appearing in both reports,
  resolves conflicts, and proposes a single prioritized list of next actions.
```

#### 9. Design principles

List at the end (5–8 bullet points):
- Default: do less. Load depth only when there is evidence.
- Prefer predictable, testable rules over clever heuristics.
- Every selection decision must be explainable in one sentence.
- Do not delegate small tasks, even if multiple skills match.
- Keep the selected skill set small. More skills = more context = slower responses.
- Cap subagents at 2–3 by default. More is not better.
- Subagents return summaries. The main agent stays responsible for the final answer.
- This is an evolving design. See docs/ai/DECISIONS.md for ADRs that affect routing.
