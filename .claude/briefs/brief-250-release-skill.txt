## Task: feat(release): add shared release-management skill

### Context

This repository is a versioned configuration kit for Claude Code, Codex CLI, and Antigravity (agy).
Skills live under `skills/<name>/SKILL.md` and are tool-agnostic prose. During install, the kit
copies each skill to `.claude/skills/`, `.agents/skills/`, and `.agy/skills/` in the target project.
The routing tables in `tooling/claude/CLAUDE.md`, `tooling/codex/AGENTS.md`, and `tooling/agy/AGY.md`
tell agents which skill to load for a given task domain.

This task adds a new `release-management` skill covering SemVer bumps, VERSION file, CHANGELOG
maintenance, release readiness, and edge cases.

### Files to create

1. `skills/release-management/SKILL.md` — the canonical shared skill (tool-agnostic prose)
2. `skills/release-management/references/edge-cases.md` — companion reference for edge cases

### Skill content guidelines (skills/release-management/SKILL.md)

The SKILL.md must use this YAML front-matter format:
```yaml
---
name: release-management
description: >
  Use when preparing, checking, or cutting a release. Covers SemVer bump rules,
  VERSION file maintenance, CHANGELOG [Unreleased] section, release readiness
  checks, tag naming, and explicit human confirmation before tagging.
---
```

Body should cover (tool-agnostic — do NOT mention Claude/Codex/Antigravity-specific tools as universal behavior):
- SemVer rules: MAJOR.MINOR.PATCH; when to bump each component.
- `VERSION` as the single source of truth for the version string.
- `CHANGELOG.md` `[Unreleased]` maintenance: what belongs there, how to promote it.
- Release readiness checklist: clean working tree, tests pass, CHANGELOG not empty, VERSION bumped.
- Tag naming convention: `vA.B.C` (e.g., `v1.2.3`).
- Explicit human confirmation required before creating or pushing any release tag.
- Never push release tags without explicit user instruction.
- Keep release changes scoped and reviewable (one PR, one concern).

### Edge cases reference (skills/release-management/references/edge-cases.md)

Cover at minimum:
- Missing or malformed `VERSION` file (file not found, empty, not in `A.B.C` format).
- Missing or malformed `CHANGELOG.md` `[Unreleased]` section.
- Release tag already exists remotely or locally.
- Dirty working tree before release (uncommitted changes).
- `HEAD` changed after verification but before tag creation.
- Branch is not the expected release branch (e.g., not `master` or `main`).
- Major or minor bump requested without explicit user confirmation.
