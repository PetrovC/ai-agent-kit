BRIEF — Issues #421 + #422: Model selection policy + automatic model switching
==============================================================================

## Context

The ai-agent-kit repository already has `docs/ai/MODEL_ROUTING.md` which gives
general guidance on model tiers.  Issues #421 and #422 ask for:

- (#421) A formal, structured model selection policy as both a human-readable
  doc and a machine-readable YAML file.
- (#422) A pure-Python CLI script (`scripts/select-model.py`) that reads the
  policy and recommends a model tier + provider for a given task, with reasons,
  fallbacks, and a confirmation flag.

This mirrors how `scripts/select-skills.py` was built in Phase 4.

## Start here

```bash
git checkout master
git pull origin master
git checkout -b agent/agy/claude-sonnet-4-6/feat/model-selection-policy
```

## Purpose

## Provider tier mapping

## Fallback rules
- If preferred provider unavailable, try fallbacks in listed order.
- Default fallback order: claude → codex → antigravity.
- Fallback uses the same tier.

## Cost-control guidance
Bullet list: prefer fast for ...; prefer high_reasoning for ...;
a good prompt + narrow scope often beats a stronger model with vague scope.

## Machine-readable policy
One-liner pointing to config/model-policy.yaml.

## select-model.py usage
Code block showing CLI usage and example output (see #422).

### 3. `scripts/select-model.py` (NEW)

Pure Python 3, stdlib only (no third-party packages).

#### CLI interface

```
python3 scripts/select-model.py --task "TEXT" [OPTIONS]

Options:
  --task TEXT          Task description (required)
  --risk LEVEL         low | medium | high | critical  (default: medium)
  --context-size SIZE  small | medium | large           (default: medium)
  --provider PROV      claude | codex | antigravity | any  (default: any)
  --json               Output JSON instead of plain text
  --debug              Print scoring details to stderr
  --dry-run            Alias for --json (kept for test parity with select-skills)
```

#### Output (plain text, default)

```
Task intent:  implementation
Base tier:    balanced
Risk bump:    none (risk=medium)
Context bump: none (context=medium)
Final tier:   balanced

Recommended:
  Provider:  claude
  Model:     claude-sonnet-4-6
  Tier:      balanced
  Confirm:   no

Fallbacks:
  codex        balanced  gpt-5.5 / reasoning_effort=medium
  antigravity  balanced  claude-sonnet-4-6
```

#### Output (--json)

```json
{
  "intent": "implementation",
  "risk": "medium",
  "context_size": "medium",
  "base_tier": "balanced",
  "final_tier": "balanced",
  "recommended_model": {
    "provider": "claude",
    "tier": "balanced",
    "model": "claude-sonnet-4-6"
  },
  "reason": "implementation task; balanced tier; no risk/context bumps",
  "requires_confirmation": false,
  "fallbacks": [
    {"provider": "codex", "tier": "balanced", "model": "gpt-5.5",
     "reasoning_effort": "medium"},
    {"provider": "antigravity", "tier": "balanced", "model": "claude-sonnet-4-6"}
  ]
}
```

#### Implementation notes

1. Find the policy file: try `Path.cwd() / "config" / "model-policy.yaml"`, then
   `Path(__file__).parent.parent / "config" / "model-policy.yaml"`.  If not
   found, print a clear error and exit 1.

2. Parse YAML without a third-party library.  The policy file uses a small,
   predictable subset of YAML (no anchors, no multi-document streams).  Write
   a minimal parser that handles:
   - Top-level keys
   - Nested dicts (2-space indent)
   - Lists of strings / dicts (the `intents` array)
   - Quoted and unquoted scalar values

   Since the policy YAML is controlled by this repo, the parser only needs to
   handle exactly the syntax in `config/model-policy.yaml`.  Alternatively,
   use `json` — note that YAML is a superset of JSON, but the policy file uses
   YAML syntax so a pure-JSON parser won't work.  Best approach: use Python's
   built-in `re` + line-by-line parsing; do NOT pull PyYAML.

   Actually, the safest approach without a YAML library: write the policy in
   a JSON-compatible subset (only double-quoted strings, no block scalars, no
   flow scalars with unquoted values) — BUT the existing `model-policy.yaml`
   above uses YAML-only features (unquoted strings, inline dicts).

   **Resolution**: read the YAML file and convert it to JSON-safe Python dicts
   using a hand-written 60–80 line parser that handles:
   - Scalars: unquoted string, double-quoted string, integer, inline dict
     `{key: "value", key2: "value2"}`
   - Lists: `  - item` and `  - {key: value}` forms
   - Dict keys at each indent level
   - Comments (`#`) on their own line or inline

   The YAML in `config/model-policy.yaml` is deliberately constrained to make
   this feasible.  Add a `# Parser constraint: no YAML anchors, no multiline
   block scalars` comment to model-policy.yaml.

3. Intent classification: iterate `intents` in policy order; for each intent,
   check if any keyword (case-insensitive) is a substring of `--task`.  Use
   the first match.  If no keyword matches, fall back to `implementation`
   (balanced tier).

4. Apply risk bumps: look up `risk_bumps[risk_level]` (default 0 if key absent).
   Clamp to the last tier in `tier_order`.

5. Apply context-size bumps: look up `context_size_bumps[context_size]`
   (default 0).  Clamp to the last tier.

6. For `--provider any`: output recommended (claude first) + fallbacks (codex,
   antigravity) at the same tier.  For a specific `--provider`: output just
   that provider; fallbacks cycle through the other two.

7. Confirmation: look up `confirmation_policy[final_tier]` from policy.

8. Exit 0 on success.  Exit 1 on policy file not found or parse error.

### 10. `tests/bats/model-routing.bats` (NEW)

```bash
#!/usr/bin/env bats
# Regression tests for scripts/select-model.py model routing.

load 'bats_helper.bash'

setup() {
    aak_setup
    SELECTOR="$KIT_ROOT/scripts/select-model.py"
}

run_selector() {
    run python3 "$SELECTOR" "$@"
}

# --- basic smoke tests ---

@test "select-model: help exits 0" {
    run_selector --help
    [ "$status" -eq 0 ]
}

@test "select-model: missing --task exits non-zero" {
    run_selector --risk low
    [ "$status" -ne 0 ]
}

# --- tier routing ---

@test "select-model: docs typo -> fast tier" {
    run_selector --task "fix typo in README" --risk low
    [ "$status" -eq 0 ]
    assert_output_contains "fast"
}

@test "select-model: architecture review -> high_reasoning tier" {
    run_selector --task "review the service layer architecture" --risk medium
    [ "$status" -eq 0 ]
    assert_output_contains "high_reasoning"
}

@test "select-model: security audit -> high_reasoning tier" {
    run_selector --task "security review of authentication code" --risk high
    [ "$status" -eq 0 ]
    assert_output_contains "high_reasoning"
}

@test "select-model: normal implementation -> balanced tier" {
    run_selector --task "implement user login endpoint" --risk medium
    [ "$status" -eq 0 ]
    assert_output_contains "balanced"
}

# --- risk bumps ---

@test "select-model: small fix + high risk -> balanced (bumped)" {
    run_selector --task "update config file" --risk high
    [ "$status" -eq 0 ]
    assert_output_contains "balanced"
}

@test "select-model: small fix + critical risk -> high_reasoning (double bump)" {
    run_selector --task "fix typo in README" --risk critical
    [ "$status" -eq 0 ]
    assert_output_contains "high_reasoning"
}

# --- context-size bumps ---

@test "select-model: balanced + large context -> high_reasoning" {
    run_selector --task "implement feature" --risk low --context-size large
    [ "$status" -eq 0 ]
    assert_output_contains "high_reasoning"
}

# --- confirmation policy ---

@test "select-model: high_reasoning tier requires confirmation (json)" {
    run_selector --task "architecture review" --json
    [ "$status" -eq 0 ]
    assert_output_contains '"requires_confirmation": true'
}

@test "select-model: fast tier does not require confirmation (json)" {
    run_selector --task "fix typo in README" --risk low --json
    [ "$status" -eq 0 ]
    assert_output_contains '"requires_confirmation": false'
}

# --- provider filter ---

@test "select-model: --provider codex outputs codex model" {
    run_selector --task "implement feature" --provider codex
    [ "$status" -eq 0 ]
    assert_output_contains "codex"
}

@test "select-model: fallbacks present in json output" {
    run_selector --task "implement feature" --json
    [ "$status" -eq 0 ]
    assert_output_contains "fallbacks"
}
```

Add a third step to the `routing-eval` job:

```yaml
      - name: Run model selector on all model fixtures
        run: |
          set -euo pipefail
          errors=0
          for fixture in tests/routing/fixtures/model-*.yaml; do
            name=$(basename "$fixture" .yaml)
            echo "--- model fixture: $name ---"
            task=$(python3 -c "
          import re
          lines = open('$fixture').read()
          m = re.search(r'^task: [\"\'](.*)[\"\']\$', lines, re.MULTILINE)
          print(m.group(1) if m else '')
          ")
            risk=$(python3 -c "
          import re
          lines = open('$fixture').read()
          m = re.search(r'^risk: (\w+)', lines, re.MULTILINE)
          print(m.group(1) if m else 'medium')
          ")
            ctx=$(python3 -c "
          import re
          lines = open('$fixture').read()
          m = re.search(r'^context_size: (\w+)', lines, re.MULTILINE)
          print(m.group(1) if m else 'medium')
          ")
            python3 scripts/select-model.py \
              --task "$task" --risk "$risk" --context-size "$ctx" || {
              echo "FAILED: $fixture"
              errors=$((errors+1))
            }
          done
          [[ "$errors" -eq 0 ]] || exit 1
          echo "All model fixtures passed"
```
